# MapSnap 상업화 기획 검토서

작성일: 2026-07-13 · 기준: 현재 코드베이스(master) · 작성 관점: 앱 서비스 기획자

## 요약 (TL;DR)

1. **최대 게이트는 기능이 아니라 '데이터 사용권과 원가'다.** 핵심 기능(T-Map)과 지도(네이버)가 모두
   외부 무료 키에 의존하는데, 두 API 모두 상업 서비스 전환 시 약관·요금 재확인이 선행돼야 한다.
   특히 **T-Map 경로를 GPX로 반출해 타사 앱(트랭글·Strava)에서 쓰게 하는 것**이 약관상 허용되는지가
   서비스 존립을 좌우한다. → [2장](#2-최우선-게이트--api-사용권원가-검증), [5장](#5-법무인허가정책-기획)
2. **두 번째 게이트는 키 보안.** 현재 T-Map·네이버 키가 APK(BuildConfig)에 내장되어 디컴파일로
   추출 가능하다. 무료 키일 때는 손해가 한도로 막히지만, 유료 전환 순간 "과금 폭탄" 리스크가 된다.
   상업화 = 프록시 서버 도입과 사실상 동의어다. → [7장](#7-보안부정사용-방지-기획), [8장](#8-백엔드-전환-기획)
3. **수익화 뼈대는 이미 절반 구현돼 있다.** 일 30회 한도 + 광고 시청 충전(+10회) 로직·Firebase
   사용량 추적이 코드에 존재하고 광고 SDK 실연동만 남았다. 1차 상업화는 이 구조(리워드 광고)로
   가볍게 출시하고, 구독은 계정·서버가 갖춰지는 2단계에서 도입하는 것을 권장한다. → [4장](#4-수익-모델bm-기획)

## 1. 현재 자산 진단 (상업화 관점)

### 이미 갖춘 것

| 자산 | 상업화 관점 의미 |
|------|------------------|
| 손그림→실도로 스냅→편집→GPX 파이프라인 | 코어 가치 완성. MVP 기능 요건 충족 |

| Firebase RTDB 기기별 사용량 추적 (ANDROID_ID 해시) | 익명 상태에서의 사용량 과금 기반 |

| ApiCallTracker (호출 수 실시간 집계) | 원가 지표 수집의 씨앗 — 분석 이벤트로 승격 가능 |

## 2. 최우선 게이트 — API 사용권·원가 검증

기능 개발보다 먼저 끝내야 하는 조사·계약 작업. 여기서 답이 틀어지면 이후 기획 전체가 바뀐다.

### 2-1. 확인 항목

| # | 항목 | 확인할 것 | 틀어질 경우 대안 |
|---|------|-----------|------------------|
| G1 | T-Map 보행자 API 상업 이용 | 무료 appKey의 상업 서비스 허용 여부, 유료 요금표(콜당 단가·구간할인), SLA | OSRM 자체 호스팅(OsrmResponse.kt 스텁 존재), 카카오모빌리티, Google Routes(보행자) |
| G2 | **경로 데이터 2차 이용(GPX 반출)** | T-Map 산출 경로를 파일로 내보내 제3자 앱에서 사용하는 행위의 약관 저촉 여부 | 라우팅 엔진 교체(OSM 기반이면 ODbL로 반출 자유) |
| G3 | 네이버 지도 SDK 요금 | NCP Maps 모바일 Dynamic Map 무료 한도·초과 단가 (요금 개편 이력 있어 최신 표 기준) | 지도 타일도 OSM 계열(MapLibre)로 교체 |
| G4 | Firebase 무료 한도 | RTDB 동시접속·저장량·전송량 Spark 한도, 초과 시 Blaze 단가 | 사용량 추적을 프록시 서버 DB로 이관 |
| G5 | 광고 네트워크 | AdMob 리워드 광고 정책(보상형 콘텐츠 제한), 국내 eCPM 시세 | AdFit·유니티애즈 등 미디에이션 |

**G1·G2는 법무 확인이 끝나기 전까지 상업화 착수 자체를 보류하는 하드 게이트로 둔다.**
G2가 특히 위험한 이유: 지도·경로 데이터 약관은 통상 "산출물의 저장·재배포·역기하 추출"을 제한하며,
MapSnap의 핵심 가치(GPX 내보내기)가 정확히 그 지점에 있다.

### 2-2. 원가 모델 (기획 산출물: 원가 시뮬레이션 시트)

계산 구조만 정의한다. 단가는 G1 확인 후 채운다.

```
월 API 원가 = MAU × 인당 월 스냅 횟수 × 스냅당 평균 청크 수 × 콜당 단가
```

- 스냅당 평균 청크 수: 현재 설계상 1~3콜(waypoint 19개 = 최대 3청크). ApiCallTracker 실측으로 보정.
- 예시 가정(단가 미정이라 자리표시): MAU 10,000 × 월 40스냅 × 2콜 = 월 80만 콜.
  콜당 1원이면 월 80만 원, 5원이면 400만 원 — **단가가 BM 선택(광고 vs 구독)을 결정한다.**
- 광고 수익 대조: 리워드 시청 1회 = +10회 충전 ≈ 10~30스냅분 원가. 국내 리워드 eCPM을
  1~3만 원(시청당 10~30원)으로 가정하면 **충전 10회의 API 원가가 시청 단가보다 낮아야 성립**.
  → 충전량(+10)·일 한도(30)는 단가 확정 후 재산정할 변수로 취급 (현재 하드코딩, 10장 리모트 컨피그와 연계).

## 3. 제품 포지셔닝 기획

### 3-1. 타겟 세그먼트 (우선순위순)

1. **코스 설계 러너** — 러닝 전 코스를 직접 짜서 트랭글·Strava·가민에 넣는 사용자. 기존 도구(웹 기반 gpx.studio, Strava 루트빌더)는 모바일에서 손으로 그리는 UX가 없다.
2. **GPX 아트 제작자** — 글자·그림 모양 경로. 왕복 획 보존 로직이 이미 이 유즈케이스를 정조준하고 있다. 소수지만 바이럴 확산력(SNS 인증)이 큼.
3. **크루 리더·코치** — 매주 코스를 만들어 멤버에게 배포. 저장/공유 기능의 핵심 수요층.
4. (확장) 등산·걷기 모임, 플로깅 코스 설계.

### 3-2. 가치 제안 문장 (안)

> "지도에 손가락으로 그리면, 실제 길로 다듬어진 러닝 코스가 된다 — 그대로 내 러닝 앱에."

### 3-3. 경쟁 분석 (기획 산출물: 경쟁 분석서)

| 서비스 | 방식 | MapSnap 대비 약점 |
|--------|------|-------------------|
| Footpath Route Planner | 손그림 스냅 (가장 유사, 글로벌) | 국내 보행자 도로망 정밀도(골목·공원길) 열세 추정 — 실측 비교 필요 |
| Strava Route Builder | 포인트 클릭 | 모바일 손그림 UX 없음, 구독 종속 |
| gpx.studio / plotaroute | 웹 기반 편집 | 모바일 완결 불가 |
| 트랭글·램블러 (국내) | 기록 중심 | 코스 '설계' 도구가 아님 — 경쟁이 아니라 **내보내기 대상 생태계** |

차별화 축: **① 국내 보행자망 스냅 정밀도 ② 모바일 손그림 완결 UX ③ GPX 아트 특화(왕복 획 보존)**.
액션: Footpath와 동일 지역 10개 코스 실측 비교 문서 작성 — 마케팅 소재 겸 품질 검증.

### 3-4. 네이밍·상표

'MapSnap'은 일반명사 조합이라 상표 등록 가능성·기존 서비스 충돌(해외 동명 제품 존재 가능) 확인 필요.
**스토어 등록 후 리브랜딩은 비용이 크므로 Phase 0에서 결정**한다. 산출물: 상표 검색 결과서, 후보명 3안.

## 4. 수익 모델(BM) 기획

### 4-1. 옵션 비교

| 옵션 | 구조 | 장점 | 단점 |
|------|------|------|------|
| A. 리워드 광고형 (현행 연장) | 일 N회 무료 + 광고 시청 충전 | 구현 잔량 최소(SDK 연동만), 원가-수익 자동 연동, 무료 사용자도 수익화 | ARPU 낮음, 광고 시청 UX 마찰 |
| B. 구독형 | 무료 체험 + 월 구독(무제한) | ARPU·예측가능성 높음 | 계정·서버·결제 검증 선행 필요, 국내 유틸 구독 전환율 낮음 |
| C. 하이브리드 (권장) | 무료(한도+광고충전) / 프로(구독) | A로 즉시 출시 → B를 나중에 얹는 단계 전략과 일치 | 한도·가격 밸런싱 설계 필요 |

### 4-2. 권장안: C를 2단계로

- **1단계 (Phase 1)**: 현행 구조에 AdMob 리워드만 실연동. 계정·서버 없이 출시 가능.
- **2단계 (Phase 2)**: 프로 구독 도입. 후보 혜택 —
  - 스냅 무제한(공정사용 상한 명시 — 원가 방어)
  - 광고 제거
  - 클라우드 경로 동기화(기기 변경 대응)
  - 고도 프로파일(`<ele>`) — 유료 DEM API 원가가 들므로 구독 전용이 자연스러움 (implementation_plan.md에서 보류한 항목의 수익화 재활용)
  - GPX 고급 옵션(타임스탬프·페이스)은 이미 무료로 나간 기능이므로 **회수하지 않는다** (기존 사용자 반발 리스크).
- 가격 가설: 월 2,900~4,900원 / 연간 40% 할인. 출시 후 스토어 가격 실험으로 검증.

### 4-3. 페이월 UX (기획 산출물: 페이월 화면 기획서)

- **한도 소진(Blocked) 시점이 유일한 전환 접점**이다. 현재 광고 다이얼로그가 뜨는 자리에
  "광고 보고 +10회 / 프로로 무제한" 2택 페이월을 설계한다.
- 잔여 횟수 카드(우측 상단, 이미 구현)는 소진 전 예고 접점 — 5회 이하 경고 시 프로 배지 노출 검토.
- 원칙: 그리기 도중에는 절대 끊지 않는다(스냅 시도 시점에만 게이트). 코어 UX 보호가 LTV보다 우선.

## 6. 계정·데이터 기획

- **현행 한계**: 기기 해시 기반이라 ① 기기 변경 시 저장 경로·충전분 소실 ② 구매(구독) 귀속 불가
  ③ 공기계 다수 확보 시 한도 우회 가능.
- **도입 시점**: 구독과 동시(Phase 2). 그 전에 계정을 강제하면 마찰만 늘고 얻는 게 없다.
  1단계 출시는 익명 유지.
- **설계 방향**:
  - Google 로그인 우선(Play 결제 귀속과 자연 연결) + 카카오(국내 보조). Firebase Auth 활용 시 기존 RTDB와 접합 용이.
  - 익명(기기 해시) → 계정 데이터 병합 시나리오 필수: 로컬 SharedPreferences 저장 경로를 계정 클라우드로 이관하는 마이그레이션 플로우.
  - 비로그인 사용은 계속 허용(코어 기능 게이트 금지) — 로그인은 '동기화·구독' 대가로만.
- 산출물: 계정 체계 설계서, 익명→계정 데이터 이관 시나리오, 로그인 유도 UX 기획.

| # | 취약점 | 현황 | 대응 |

| S1 | **API 키 APK 내장** | T-Map·네이버 키가 BuildConfig로 컴파일됨 — 디컴파일 추출 가능 | 근본: 프록시 서버(8장). 임시: 키 사용량 알림 설정, 네이버 키의 앱 서명·패키지 제한, T-Map 키 IP/리퍼러 제한 가능 여부 확인 |
| S2 | Firebase RTDB 규칙 | R-12 미확인 — 열려 있으면 dailyCount 위조로 한도 우회 | 콘솔에서 `.validate` 규칙(+1 증가만 허용) 적용, **Phase 1 출시 전 필수** |
| S3 | 클라이언트 한도 로직 | 한도 판정이 클라이언트에 있어 개조 APK로 우회 가능 | 무료 키 동안은 수용(손해 상한 = 무료 쿼터), 유료 전환 시 프록시에서 서버 판정 |
| S4 | 결제 위변조 (Phase 2) | — | Play Integrity API + 영수증 서버 검증 |

원칙: **무료 키 단계에서는 S2까지만 막고, 유료 계약 체결 전에 S1·S3(프록시)을 완료**한다.
유료 키를 APK에 넣는 순간이 가장 위험한 시기다.

## 8. 백엔드 전환 기획

- **왜 필요한가**: 키 은닉(S1), 서버측 한도 강제(S3), 구독 검증(S4), 경로 클라우드 저장(6장), 라우팅 엔진 교체 유연성(G1 대안) — 다섯 요구가 전부 같은 곳을 가리킨다.
- **구조 (안)**: 경량 프록시 — `클라이언트 → 프록시(인증·한도·키) → T-Map`.
  - 클라이언트는 `RouteRepository` 인터페이스 뒤에 이미 추상화되어 있어 **구현체 교체만으로 전환 가능** (아키텍처가 이미 준비됨 — 전환 비용 낮음).
  - 후보: Cloud Run / Cloud Functions (Firebase 생태계 연속성) 등 종량제 서버리스. 월 수십만 콜 규모 견적 산출.
  - 프록시 도입 시 사용량 추적도 RTDB 클라이언트 쓰기 → 서버 기록으로 이동해 S2·S3이 함께 해소된다.
- **시점**: Phase 2 초입(유료 API 계약 직전). Phase 1은 무료 키+클라이언트 직call 유지.
- 산출물: 백엔드 요구사항 정의서(API 스펙·한도 정책·인증), 인프라 비용 견적서.

## 9. 스토어 출시·ASO 기획

- **Play Console 요건**: 개인정보처리방침 URL, 데이터 보안 양식(위치·기기ID 수집 신고), `AD_ID` 선언, 콘텐츠 등급 설문, 위치 권한 사용 사유 영상/설명.
- **심사 리스크**: 포그라운드 위치 권한은 "현재 위치 표시" 사유로 통과 무난. 백그라운드 위치는 쓰지 않음을 명확히.
- **ASO 키워드 (안)**: 러닝 코스 만들기, GPX, 경로 그리기, 러닝 지도, GPX 아트, 코스 설계, 마라톤 연습 코스.
- **스크린샷 스토리(5컷)**: ① 손으로 그린다 ② 실제 길로 스냅된다 ③ 마커로 다듬는다 ④ GPX로 내보낸다 ⑤ 트랭글/Strava에서 달린다.
- **출시 전략**: 비공개 테스트(러닝 크루 20~50명) → 오픈 베타 → 정식. Play 신규 개인 개발자 계정은 비공개 테스트 요건(테스터 수·기간)이 있으므로 일정에 반영.
- 산출물: 스토어 리스팅 킷(문안·스크린샷·그래픽), 데이터 보안 양식 응답지, 베타 테스터 모집 계획.

## 10. 운영·분석 기획

- **도구 스택 (Phase 1 필수)**: Crashlytics(크래시), GA4 for Firebase(행동), Remote Config(정책값).
  - Remote Config 대상 1순위: 일 한도(30)·충전량(10)·페이월 문안 — 현재 하드코딩(R-10)이라 상수화 작업과 함께 서버 조정 가능하게. **원가 비상시 앱 업데이트 없이 한도를 조일 수 있어야 한다.**
- **이벤트 택소노미 (초안)**: `draw_start`, `snap_request(waypoints, chunks)`, `snap_success/fail(reason)`,
  `spur_removed`, `limit_blocked`, `ad_reward_complete`, `gpx_export(timestamps_on)`, `route_save/load`,
  `edit_apply(clusters, calls)` — ApiCallTracker 집계를 이벤트로 승격하면 사용자당 원가 콜 수가 대시보드에서 바로 나온다.
- **CS**: 스토어 리뷰 응대 SLA(48h), 인앱 문의(메일 폼), FAQ(도움말 다이얼로그 확장 — 온보딩 자산 재활용).
- **품질 게이트**: 실기기 확인 필요 항목(walkthrough.md에 누적된 것들)을 출시 전 QA 시나리오로 정리 — 스냅 파이프라인은 실측 지역 의존성이 커서 지역별 QA 코스 목록(강남 골목·공원·강변·아파트 단지) 문서화.
- 산출물: 이벤트 정의서, 운영 대시보드 기획, QA 시나리오·지역 코스 목록, CS 매뉴얼.

## 11. KPI 기획

- **북극성 지표**: 주간 완성 경로 수 (스냅 성공 후 '저장 또는 GPX 내보내기'까지 도달한 경로).
  — 설치나 DAU가 아니라 "코스를 완성해 실제로 쓰는" 행동이 가치 실현 순간이다.
- **퍼널**: 설치 → 첫 그리기 시작 → 첫 스냅 성공 → 첫 내보내기/저장 → 7일 내 재방문.
  각 단계 이탈률 측정이 GA4 이벤트 설계(10장)의 존재 이유.
- **수익**: ARPDAU, 리워드 광고 시청률(노출 대비), (Phase 2) 체험→구독 전환율, 월 해지율.
- **원가**: 사용자당 일 평균 API 콜, 스냅당 평균 청크 수(최적화 회귀 감지), 한도 도달률.
  - 한도 도달률은 이중 의미 — 너무 높으면 UX 불만, 너무 낮으면 페이월 접점 부재. 목표 밴드(예: DAU의 5~15%)를 정하고 Remote Config로 한도를 조정한다.

## 12. 단계별 로드맵

| 단계 | 목표 | 주요 작업 | 게이트(선행조건) |
|------|------|-----------|------------------|

각 Phase 전환 시점에 킬 크라이테리아를 함께 정한다 — 예: Phase 1 출시 3개월 후
주간 완성 경로 수가 목표 미달이고 리워드 시청률이 X% 미만이면 Phase 2 투자를 보류하고 포지셔닝 재검토.
